# Workspace Symlink Strategy

## The Problem

Every OS reinstall means rebuilding `/home` from scratch. Some things are easy to
reinstall (packages, flatpaks), but configs and project data that live in `~`
get wiped. The standard approach ‚Äî backup/restore ‚Äî is fragile: you forget
something, the backup is stale, or you restore garbage you should have cleaned up.

On this system, `/mnt/workspace` is a separate NVMe that never gets formatted.
It already held model files, llama.cpp builds, and project directories. But
critical dotfiles and tool configs still lived in `/home` and died with every
reinstall.

## The Goal

A single-command restore after a full wipe (``/, `/home`, everything):

```bash
./link-workspace.sh --apply
```

Only 6 things are worth persisting ‚Äî everything else regenerates fresh.

## What We Persist

| Path | Why |
|------|-----|
| `~/.ssh` | SSH keys, `known_hosts`, `authorized_keys` ‚Äî irreplaceable |
| `~/.librewolf` | Browser profiles, bookmarks, sessions, extensions |
| `~/.openclaw` | Legacy OpenClaw config (migrated to `~/.config/opencode/opencode.json`) |
| `~/.opencode` | OpenCode IDE agent config and state |
| `~/dotfiles` | This repo ‚Äî shell configs, scripts, docs, gitconfig |
| `~/openclaw` | OpenClaw project workspace directory |

Everything else in `/home` gets regenerated by applications on first run:
`.cache`, `.config/*`, `.local/share/*`, `.steam`, `.wine`, `Downloads`,
`Documents`, etc.

For example, after a fresh install and running `--apply`:
- **Firefox/Librewolf** ‚Äî creates `.cache/librewolf`, `.config/librewolf`
  on first launch (profiles live in `.librewolf` which is symlinked, so
  bookmarks and sessions persist)
- **Steam** ‚Äî creates `.steam`, `.local/share/Steam` on install
- **Discord** ‚Äî creates `.config/discord`, `.local/share/Discord` on first run
- **Zsh** ‚Äî creates `.zshrc` via bootstrap.sh, which sources dotfiles
- **OpenCode** ‚Äî creates `.config/opencode`, `.local/share/opencode`
  (config + DB live on workspace via `.opencode` symlink)

## How It Works

The script (`scripts/link-workspace.sh`) does three things:

1. **Merge** ‚Äî If a directory exists in `/home` but not on workspace, it's
   rsynced to workspace first (home ‚Üí workspace, workspace version wins
   conflicts).
2. **Link** ‚Äî The home directory is replaced with a symlink to workspace.
3. **Skip** ‚Äî Already-correct symlinks are left alone.

On a fresh install, `/home` is empty so there's nothing to merge ‚Äî it just
creates the 6 symlinks and you're done. Workspace was the canonical copy all
along.

### Quick Start: After a Fresh OS Install

```bash
# 1. Mount workspace drive (or restore from backup)
sudo mount /mnt/workspace

# 2. Clone dotfiles (workspace is the canonical copy, so clone there)
git clone git@github.com:InnerTic/dotfiles.git /mnt/workspace/dotfiles

# 3. Create the 6 symlinks (merge step is a no-op on empty /home)
/mnt/workspace/dotfiles/scripts/link-workspace.sh --apply

# 4. Run bootstrap ‚Äî this creates ~/.zshrc, ~/.gitconfig, etc. from dotfiles
/mnt/workspace/dotfiles/bootstrap.sh

# 5. Verify everything landed
/mnt/workspace/dotfiles/scripts/link-workspace.sh --status

# Done. Apps will regenerate their own ~/.config, ~/.cache, etc. on first use.
# Profiles, keys, and projects are already in place from workspace.
```

**Order matters:** Run `link-workspace.sh` **before** `bootstrap.sh`. The symlinks
need to exist first so bootstrap's symlinks (e.g. `~/.zshrc` from dotfiles) land
correctly relative to the rest of the workspace layout. Bootstrap handles config
files; `link-workspace.sh` handles directories.

## Idempotency

Safe to run multiple times. Already-linked paths are detected and skipped.
Wrong targets are corrected. A `validate_workspace()` check prevents accidental
setup if workspace isn't mounted or is on the same filesystem as `/home`.

## The Alternative We Chose Not To Use

- **Full `/home` backup** ‚Äî Too much garbage, too easy to restore cruft.
  Most of `~/.config`, `~/.cache`, `~/.local/share` is autogenerated and
  should start fresh.
- **Bind mounts in fstab** ‚Äî Works for media dirs (Documents, Downloads) but
  creates boot-time dependency and makes perms harder to fix.
- **Dotfiles-only** ‚Äî We already had that. Tool configs and project data were
  the missing piece.
